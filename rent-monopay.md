

Нижче повний поточний флоу оренди SUPboard, саме так як він зараз реалізований у коді, з прямими переходами по файлах.

**1. Запуск усіх підсистем**
1. При старті бота запускаються 3 паралельні задачі: polling Telegram, таймер, webhook сервер Monobank у bot.py.
2. Підключення роутерів і порядок обробки подій тут: bot.py.
3. Таймер запускається кожні 15 секунд у bot.py.
4. Monobank webhook сервер запускається тут: bot.py.

**2. Вхід користувача і доступ до оренди**
1. Кнопка Орендувати має callback rent у kb.py.
2. На start перевіряється, чи користувач зареєстрований, інакше показується меню реєстрації у start.py.
3. Кнопка Реєстрація запускає FSM реєстрації у req.py.
4. Після підтвердження реєстрації користувач записується в БД через req.py.

**3. Початок оренди**
1. Callback rent обробляється у rent.py.
2. Бот показує активні станції, далі вибір станції через station_id у rent.py.
3. Після вибору станції показується список доступних комірок у rent.py.
4. Список доступних комірок бере з БД тут: db.py.

**4. Вибір комірок і створення резерву**
1. Мультивибір комірок обробляється callback cell_ у rent.py.
2. Після натискання Підтвердити callback done_cells у rent.py.
3. Для кожної вибраної комірки створюється запис оренди зі статусом Резервація і таймером 60 тікiв через db.py.
4. Оновлення комірки в статус Резервація робиться в тому ж блоці у rent.py.

**5. Вибір тривалості і розрахунок ціни**
1. Вибір часу оренди обробляється у rent.py.
2. Для кожної комірки знаходиться тариф і ціна, після чого зберігаються base_time та subscription через db.py.
3. Після розрахунку оренда переводиться у статус Очікує оплату і таймер 20 тiкiв через rent.py.

**6. Створення інвойсу Monobank**
1. Інвойс створюється викликом сервісу у rent.py.
2. Формування payload та виклик Monobank create invoice у payment_service.py.
3. Запис транзакції в payment_transactions у db.py.
4. Фактичний HTTP виклик до Monobank endpoint invoice/create у monobank_client.py.
5. Посилання на оплату повертається користувачу кнопкою URL у rent.py.

**7. Що з webHookUrl і redirect**
1. Redirect URL береться з MONO_REDIRECT_URL через конфіг у config.py.
2. webHookUrl формується у payment_service.py.
3. Зараз webHookUrl додається в payload тільки якщо валідний, дивись payment_service.py.

**8. Підтвердження оплати (автоматично)**
1. Webhook endpoint піднімається тут: webhook_server.py.
2. Реєстрація POST маршруту на MONO_WEBHOOK_PATH: webhook_server.py.
3. Перевірка підпису X-Sign і обробка payload: webhook_server.py.
4. Криптоперевірка підпису через публічний ключ Monobank: payment_service.py.
5. Після webhook статус нормалізується і оновлюється транзакція: payment_service.py, db.py.

**9. Якщо webhook не прийшов**
1. Є fallback reconcile цикл раз у 90 секунд: webhook_server.py.
2. Він опитує pending/processing транзакції: payment_service.py, db.py.
3. Для кожної транзакції робить status API Monobank: monobank_client.py.

**10. Що стається при успішній первинній оплаті**
1. В _mark_paid для initial оренди переводяться у Очікування відкриття, таймер 20 тiкiв, pay_1 = OK: payment_service.py, db.py.
2. Користувач після цього бачить оренду в Мої оренди, бо туди потрапляють Очікування відкриття або Оренда: db.py.
3. Кнопки Мої оренди і Завершити оренду формуються тут: start.py.

**11. Старт фактичної оренди і відкриття комірки**
1. Клік по openLocker обробляється тут: start.py.
2. Відкриття замка йде через Home Assistant switch turn_on у start.py.
3. Якщо статус був Очікування відкриття, оренда переводиться в Оренда і таймер стає base_time * 4 у start.py.
4. Через 15 секунд виконується auto turn_off замка у start.py.

**12. Таймерна логіка кожні 15 секунд**
1. Центральний обробник таймера: utilits_funk.py.
2. Для Резервація та Очікує оплату: при 0 скасовує резерв і звільняє комірку, інакше декрементує таймер: utilits_funk.py.
3. Для Очікування відкриття: коли лишається 1 тік, автоматично стартує Оренда: utilits_funk.py.
4. Для Оренда: на 20 тiках попередження за 5 хв, на 1 тiку повідомлення що час вийшов: utilits_funk.py.

**13. Завершення оренди**
1. Кнопка finishRent веде в FSM завершення у finishRent.py.
2. Користувач надсилає фото комплекту, воно зберігається в rent і пересилається адмінам: finishRent.py.
3. Після натискання Кінець оренди перевіряється датчик дверей через HA: finishRent.py.
4. Якщо двері закриті і перевищення невелике, оренда завершується одразу: finishRent.py.
5. Якщо є доплата, створюється topup інвойс Monobank: finishRent.py, payment_service.py.
6. Після success topup платіж підтверджується автоматично і оренда завершується у _mark_paid: payment_service.py.

**14. Важливі нюанси поточної реалізації**
1. FSM стан waiting_payment_proof практично не використовується в новому флоу, бо після видачі платіжного URL стан очищається в rent.py, а обробники для waiting_payment_proof сидять у rent.py.
2. Отже основне підтвердження оплати зараз повністю через webhook + reconcile, а не через фото квитанцій.

Якщо хочеш, наступним кроком розпишу це у вигляді компактної таблиці переходів станів саме по status полю в rent, щоб ти міг швидко дебажити будь-який кейс по одному запису з БД.

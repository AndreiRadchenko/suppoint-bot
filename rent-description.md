# Rent Flow Description

## 1) Where customer is allowed to open inventory cell

The permission point is in `openLocker` callback handler:

- `handlers/start.py:655-693` (`@router.callback_query(F.data.startswith('openLocker:'))`)
- The bot first loads only current user's active rents via `db.get_all_my_rent(tg_id=call.from_user.id)` and then checks selected `rent_id` against that list (`handlers/start.py:660-668`).
- If matched, bot sends unlock command through Home Assistant switch (`await switch_on_handler(locker[4])`, `handlers/start.py:669`).
- If rent status is `Очікування відкриття`, the handler starts rental immediately:
  - `db.rent_update_status_and_timer('Оренда', int(rent[4]) * 4, rent[0])` (`handlers/start.py:684-687`)
  - `db.locker_status('Оренда', rent[3])` (`handlers/start.py:687`)
- Auto-lock is scheduled after opening: `asyncio.create_task(delayed_switch_off(locker[4]))` (`handlers/start.py:689`).

Related HA helpers:
- `toggle_switch`: `handlers/start.py:871-880`
- `switch_on_handler`: `handlers/start.py:883-891`
- `delayed_switch_off`: `handlers/start.py:899-901`

## 2) High-level runtime wiring

- Bot + scheduler run concurrently in `bot.py`:
  - Routers included in order: `rent`, `finishRent`, `req`, `error_report`, `start` (`bot.py:32-36`)
  - Scheduler executes `timer()` every 15 seconds (`bot.py:15-23`)

Timer implementation:
- `helper/utilits_funk.py:10-59`

## 3) Booking flow (user side)

FSM states are defined in `RentBoard`:
- `choosing_station`, `choosing_cells`, `choosing_rent_time`, `waiting_payment`, `waiting_payment_proof` (`handlers/rent.py:18-23`)

### Step A: Start booking
- Callback `rent` -> show active stations and set FSM to `choosing_station` (`handlers/rent.py:125-146`)
- DB read: `db.get_all_active_stations()` (`handlers/rent.py:130`)

### Step B: Choose station and lockers
- Station callback (`station_...`) stores `station_id`, initializes `selected_lockers` (`handlers/rent.py:152-159`)
- Lockers shown from DB (`db.get_all_available_lockers`, `db.get_inventory_kit_by_locker_and_station_id`) in `show_locker_selection` (`handlers/rent.py:26-70`)
- Locker toggle callback (`cell_...`) updates FSM data (`handlers/rent.py:165-181`)

### Step C: Confirm selected lockers
- `done_cells` callback creates draft rent rows with reservation status (`handlers/rent.py:186-223`)
- DB writes:
  - `db.add_base_rent(..., 'Резервація', 60)` (`handlers/rent.py:203`)
  - `db.locker_status('Резервація', locker_id)` (`handlers/rent.py:204`)
- FSM -> `choosing_rent_time` (`handlers/rent.py:206`)

### Step D: Select duration and compute price
- Time callback (`time_...`) handles quick and extended durations (`handlers/rent.py:228-267`)
- For concrete duration (`handlers/rent.py:268-323`):
  - DB reads:
    - `db.get_inventory_kit_by_locker_and_station_id` (`handlers/rent.py:281`)
    - `db.get_tariff_by_data` (`handlers/rent.py:284`)
  - DB write:
    - `db.update_price_and_time_in_rent(...)` (`handlers/rent.py:287`)
- Payment URL generated by `generate_bank_qr_url` (`handlers/rent.py:72-102`, used at `handlers/rent.py:305-311`)
- FSM -> `waiting_payment`, then `waiting_payment_proof` (`handlers/rent.py:303`, `322`)

### Step E: User uploads payment proof
- Handler: `@router.message(RentBoard.waiting_payment_proof, F.photo | F.document)` (`handlers/rent.py:329-364`)
- DB writes for each selected locker:
  - `db.locker_status('Перевірка оплати', locker_id)` (`handlers/rent.py:355`)
  - `db.update_status_and_timer_for_rent(..., 'Перевірка оплати', 0, file_type, file_id, 'Резервація')` (`handlers/rent.py:356`)
- Admins are notified (`handlers/rent.py:358-359`)
- FSM cleared (`handlers/rent.py:361`)

## 4) Admin payment review and approval

### Review queue
- Callback `check_pay` reads pending records (`handlers/start.py:116-167`)
- DB read: `db.get_rent_on_inspection()` (`handlers/start.py:120`)
- For each rent, admin receives payment media and inline actions:
  - `rentOk:<id>`
  - `reSend:<id>`
  - `rentNot:<id>`
  (`handlers/start.py:149-163`)

### Admin decision branches

1. Approve (`rentOk`): `handlers/start.py:304-324`
- DB writes:
  - `db.rent_update_status_and_timer('Очікування відкриття', 20, rent_id)` (`handlers/start.py:310`)
  - `db.locker_status('Очікування відкриття', rent[3])` (`handlers/start.py:311`)
  - `db.rent_update_pay_1_status(rent_id)` (`handlers/start.py:312`)
- User gets instructions to open locker from `Мої оренди`.

2. Request re-send (`reSend`): `handlers/start.py:287-300`
- DB write:
  - `db.rent_update_status_and_timer('Повторний запит', 20, rent_id)` (`handlers/start.py:298`)

3. Reject (`rentNot`): `handlers/start.py:270-283`
- DB writes:
  - `db.rent_update_status('Відміна адміністратором', rent_id)` (`handlers/start.py:279`)
  - `db.locker_status('Доступна оренда', rent[3])` (`handlers/start.py:280`)

## 5) Start of rental after approval

There are two start paths:

1. Manual open by customer:
- `my_rent` menu builds `openLocker:<rent_id>` buttons (`handlers/start.py:339-361`)
- `openLocker` callback performs unlock and starts rent if status is `Очікування відкриття` (`handlers/start.py:655-689`)

2. Automatic start by scheduler if user does not open in time:
- Timer checks status `Очікування відкриття` (`helper/utilits_funk.py:37-42`)
- When timer reaches 1 tick:
  - User notified: `Розпочато оренду`
  - `db.rent_update_status_and_timer('Оренда', int(rent[4]) * 4, rent[0])` (`helper/utilits_funk.py:40`)

## 6) Active rent timer behavior

`timer()` in `helper/utilits_funk.py:10-59` handles statuses:

- `Резервація` and `Очікує оплату`: decrement timer or cancel + free locker (`helper/utilits_funk.py:16-29`)
- `Повторний запит`: decrement or cancel + free locker (`helper/utilits_funk.py:30-36`)
- `Очікування відкриття`: decrement or auto-start rent (`helper/utilits_funk.py:37-42`)
- `Оренда`:
  - at `timer == 20` send 5-minute warning (`helper/utilits_funk.py:44-46`)
  - at `timer == 1` send overtime message (`helper/utilits_funk.py:47-51`)
  - always decrement (`helper/utilits_funk.py:46,51,53`)

## 7) Finish rent flow

FSM states in `RentFinishFSM`:
- `waiting_for_photo`, `waiting_for_confirmation` (`handlers/finishRent.py:21-23`)

### Step A: User starts finish
- Callback `finishRent:<rent_id>` (`handlers/finishRent.py:83-117`)
- DB reads:
  - `db.get_rent_by_id` (`handlers/finishRent.py:95`)
  - `db.get_locker_by_locker_id` (`handlers/finishRent.py:97`)
  - `db.get_inventory_kit_by_locker_and_station_id` (`handlers/finishRent.py:98`)
- FSM -> `waiting_for_photo` (`handlers/finishRent.py:115`)

### Step B: User sends kit photo/document
- Handler `photo_received` (`handlers/finishRent.py:123-164`)
- DB write:
  - `db.rent_update_complect_photo(file_type, file_id, rent_id)` (`handlers/finishRent.py:138`)
- Admin notification with media (`handlers/finishRent.py:139-147`)
- FSM -> `waiting_for_confirmation` (`handlers/finishRent.py:161`)

### Step C: Confirm finish (door must be closed)
- Callback `confirm_rent_finish` (`handlers/finishRent.py:167-277`)
- Reads door sensor from HA:
  - `get_entity_state(sensor, config.tg_bot.ha_url, config.tg_bot.ha_token)` (`handlers/finishRent.py:174`)

If sensor is `close`:
- Load rent (`handlers/finishRent.py:176-178`)
- Branch 1: no significant overtime (`rent[13] > -20`) (`handlers/finishRent.py:180-190`)
  - DB writes:
    - `db.rent_update_status_and_timer('Завершено', 0, rent[0])` (`handlers/finishRent.py:181`)
    - `db.add_total_time(rent[4], rent[0])` (`handlers/finishRent.py:182`)
- Branch 2: overtime (`handlers/finishRent.py:191-260`)
  - Recalculate duration and tariff (`handlers/finishRent.py:196-218`)
  - If surcharge <= 0: same as normal completion (`handlers/finishRent.py:220-232`)
  - If surcharge > 0:
    - DB writes:
      - `db.rent_update_surcharge(fin_pay, rent_id)` (`handlers/finishRent.py:242`)
      - `db.rent_update_status_and_timer('Очікує доплату', 0, rent[0])` (`handlers/finishRent.py:256`)
      - `db.add_total_time(total_time, rent[0])` (`handlers/finishRent.py:257`)
    - Sends surcharge payment link (`handlers/finishRent.py:234-255`)
- Locker always returned to available:
  - `db.locker_status('Доступна оренда', rent[3])` (`handlers/finishRent.py:261`)

If sensor is not `close`:
- Finish is blocked and user is asked to close locker and retry (`handlers/finishRent.py:262-275`)

## 9) Status/state transition summary

### Business statuses in `rent` table
Typical path:
1. `Резервація` -> created after locker selection (`handlers/rent.py:203`)
2. `Перевірка оплати` -> after user uploads payment proof (`handlers/rent.py:356`)
3. Admin result:
   - `Очікування відкриття` (`handlers/start.py:310`) or
   - `Повторний запит` (`handlers/start.py:298`) or
   - `Відміна адміністратором` (`handlers/start.py:279`)
4. `Оренда` -> when locker opened or auto-start (`handlers/start.py:686`, `helper/utilits_funk.py:40`)
5. End states:
   - `Завершено` (`handlers/finishRent.py:181/221`) or
   - `Очікує доплату` (`handlers/finishRent.py:256`) and then later closed by admin surcharge flow

### FSM states
- Booking FSM (`RentBoard`): `choosing_station` -> `choosing_rent_time` -> `waiting_payment` -> `waiting_payment_proof` -> clear (`handlers/rent.py:18-23`, `145`, `206`, `303`, `322`, `361`)
- Finish FSM (`RentFinishFSM`): `waiting_for_photo` -> `waiting_for_confirmation` -> clear (`handlers/finishRent.py:21-23`, `115`, `161`, `189/231/259`)
